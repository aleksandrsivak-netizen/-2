# Независимый строгий тест алгоритма TERCOM («торком»)

Этот каталог — **внешний, независимый** испытательный стенд для ядра
`tercom_uav` (алгоритм terrain-referenced navigation). Он проверяет, что
алгоритм по карте рельефа и потоку радиовысотомера **сам**, без ГНСС,
определяет: координаты `x,y`, путевой угол (азимут), путевую скорость и
вектор скорости `vx,vy` — и честно сообщает, когда привязка ненадёжна.

«Геоштурман» (`GeoShturman/`) здесь сознательно НЕ тестируется: это
визуализация данных датчиков, а проверяемая «истина» проекта — сам алгоритм.

## Почему тест независимый

Стенд не доверяет ни одному компоненту проекта, который мог быть «подогнан»
под алгоритм. Всё, что нужно для генерации входа, реализовано заново:

| Компонент входных данных | Кто его делает в тесте | Зачем независимо |
|---|---|---|
| Рельеф (DEM) | свой спектральный генератор `make_terrain`, свой seed | не использовать `DEMGrid.synthetic` проекта |
| «Истинная» высота вдоль трассы | свой билинейный оракул `oracle_sample` | не зависеть от `DEMGrid.sample`, который применяет сам алгоритм |
| Кодирование NMEA-0183 GPGGA | свой кодировщик + свой расчёт контрольной суммы | проверить парсер алгоритма независимым кодировщиком |
| Траектория-истина, метрики ошибок | геометрия в самом стенде | алгоритм истину НЕ получает |

Алгоритму на вход подаётся **только**: карта высот, файл NMEA (радиовысоты +
метки времени), барометрическая высота и грубая подсказка скорости.
Алгоритм **не получает**: истинный азимут, точку старта, истинную скорость,
truth-CSV (`truth=None`). Все ошибки считает стенд своим кодом.

Поля широты/долготы в генерируемом NMEA **пустые** (как в примере из ТЗ:
`$GPGGA,123519.111,,,,,,,,545.4,M,46.9,M,,*47`) — это прямое доказательство,
что привязка не использует ГНСС.

## Состав

- `independent_harness.py` — основной набор из 14 сценариев + сводная таблица.
- `geonav_cli.py` — CLI с двумя режимами: `auto` (автономный сбор данных
  датчика и локализация) и `manual` (локализация по готовому NMEA-файлу).
- `_artifacts/` — сюда пишутся сгенерированные `*.nmea` и `report.txt`.

## Запуск

```powershell
# из корня клона репозитория, в venv с numpy/scipy/pandas/matplotlib/typer
.\.venv\Scripts\python.exe independent_test\independent_harness.py
```

Стенд печатает таблицу и дублирует её в `independent_test/_artifacts/report.txt`.
Код возврата `0`, если пройдены все сценарии.

### Автономный режим (борт сам понимает, где он и куда летит)

```powershell
.\.venv\Scripts\python.exe independent_test\geonav_cli.py auto --azimuth 73 --speed 55 --duration 150
```

Стенд синтезирует поток радиовысотомера по заданному курсу/скорости, затем
алгоритм — не зная истины — выдаёт координаты, путевой угол и скорость.
Истина печатается отдельной строкой только для сверки.

### Ручной ввод (готовый NMEA с борта)

```powershell
.\.venv\Scripts\python.exe independent_test\geonav_cli.py manual --nmea путь\к\board.nmea --baro 1500 --speed-hint 50
```

Можно подать реальный GeoTIFF DEM через `--dem path\to\dem.tif`
(требует установленного `rasterio`).

## Что проверяет каждый сценарий

| Сценарий | Что проверяется (по ТЗ) |
|---|---|
| `az_000…az_359` | перебор азимута 0–360°, точность направления по всем квадрантам |
| `speed_fast/slow_hint_*` | скорость как **выход** алгоритма при искажённой подсказке |
| `high_noise` | устойчивость к шуму радиовысотомера (σ=8 м) |
| `dropouts` | пропадание 25% сообщений NMEA (нерегулярная дискретизация) |
| `offset_start` | поиск сдвига вдоль луча при старте не по центру карты |
| `rate_1hz` | нижняя граница частоты сообщений по ТЗ (1 Гц) |
| `flat_terrain` | **негативный тест**: на плоском рельефе алгоритм обязан сам сообщить о ненадёжности (низкая уверенность / `ambiguous`), а не «выдумать» координаты |

## Метрики в таблице

- `d_az` — ошибка азимута, градусы;
- `d_trk` — ошибка путевого угла (из вектора `vx,vy`), градусы;
- `d_v%` — относительная ошибка путевой скорости, %;
- `pos_onl` — ошибка позиции онлайн-окна, **сверка с истиной в момент времени
  самой оценки** (последнее скользящее окно относится к t≈середине окна, это
  естественная задержка онлайн-режима, а не ошибка);
- `pos_post` — ошибка позиции постфактум по полному профилю (самый чистый «fix»);
- `conf` — самооценка уверенности; `amb` — флаг неоднозначности.

## Важные методические выводы

1. **Привязка постфактум — порядка единиц/десятков метров** на трассе ~9 км
   при 30-м DEM; азимут восстанавливается с нулевой ошибкой в пределах шага
   сетки 1°. Это и есть основной критерий корректности ядра.
2. **Онлайн-оценку нужно сверять по её собственному времени** `time_s`, иначе
   возникает мнимая «ошибка» ≈ полуокно × скорость. В первой версии стенда это
   давало ложные ~280 м — устранено.
3. **Подсказка скорости важна.** При грубо искажённой (на 20–25%) подсказке
   деградирует один из двух оценщиков (онлайн или постфактум), но второй
   остаётся точным — отказ взаимодополняющий, поэтому система всегда имеет
   достоверный выход. На реальном борту подсказку даёт воздушная скорость/ИНС,
   поэтому это крайний, состязательный вход.
4. **Плоский рельеф** алгоритм отрабатывает честно: уверенность 0, флаг
   неоднозначности — навигация по рельефу там физически невозможна, и
   алгоритм об этом сообщает, а не «врёт» координатами.
