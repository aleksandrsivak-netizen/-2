# TERCOM-навигация БПЛА

Репозиторий содержит два прототипа по кейсу terrain-referenced navigation для гражданских воздушных судов при недоступном или ненадежном GNSS.

## Роль прототипов

**GeoShturman** - основной продуктовый ответ команды на кейс. Это демонстрационный прототип с backend, dashboard и операторским сценарием: показать заказчику, как поток высотных измерений превращается в оценку положения, курса, скорости и режима достоверности. Инструкции и детали запуска находятся в [GeoShturman/README.md](GeoShturman/README.md). Файлы внутри `GeoShturman/` в этой ветке не редактировались.

**tercom_uav** - независимая эталонная batch-реализация алгоритма. Она удобна для постфактум-разбора NMEA-логов, регрессионных тестов, проверки confidence/ambiguity-логики и оценки того, какой режим поиска можно переносить на слабый бортовой вычислитель. Этот прототип проще по интерфейсу, но хорошо подходит как инженерный стенд для воспроизводимых экспериментов.

Сценарии гражданского применения, требования к воздушным судам, датчикам, вычислителю и интеграции с автопилотом описаны отдельно: [BUSINESS_SCENARIOS.md](BUSINESS_SCENARIOS.md).

## Быстрый запуск GeoShturman

GeoShturman запускается как web-приложение. Актуальная инструкция поддерживается в его собственном README:

```bash
cd GeoShturman
```

Дальше следуйте [GeoShturman/README.md](GeoShturman/README.md): там описаны Docker-запуск, локальный запуск backend и работа с dashboard/API. В корневом README эти шаги не дублируются, чтобы не расходиться с продуктовой веткой.

## Быстрый запуск tercom_uav

Рекомендуемый Python для Windows - 64-битный `py -3.13`.

```bash
cd tercom_uav
py -3.13 -m venv .venv
.\.venv\Scripts\python -m pip install -e ".[dev]"
.\.venv\Scripts\python -m pytest -q
```

Демо на синтетической DEM-карте:

```bash
.\.venv\Scripts\python -m tercom_uav.cli demo --out outputs/demo1
```

Локализация готового NMEA-лога:

```bash
.\.venv\Scripts\python -m tercom_uav.cli localize ^
  --nmea input.nmea ^
  --dem path/to/dem.tif ^
  --baro-alt 1500 ^
  --out outputs/localize1
```

Web-интерфейс `tercom_uav`:

```bash
.\.venv\Scripts\python -m tercom_uav.cli ui
```

Открыть `http://127.0.0.1:8765`.

## Известные ограничения

TERCOM не является универсальной заменой GNSS. Он работает лучше там, где профиль рельефа достаточно информативен, а хуже - над плоской или периодичной местностью.

В `tercom_uav` это обработано явно:

- `low_observability=True` выставляется, когда эталонный DEM-профиль имеет слишком малые `reference_profile_std_m` или `reference_profile_range_m`; confidence при этом ограничивается сверху независимо от gap в heatmap.
- Полностью вырожденный flat-terrain поиск возвращает no-fix результат, а не аварийное исключение.
- `coarse_to_fine` доступен через флаг, но выключен по умолчанию: на пересеченном рельефе он может совпадать с full-grid, а на гладком рельефе требует отдельной валидации.
- Скользящие окна проходят кинематический gate: физически невозможные скачки заменяются dead reckoning и помечаются как неоднозначные.
- Подсказка скорости не принимается на веру: batch-прототип перебирает несколько speed-гипотез, но на гладком периодическом рельефе одиночное окно все равно может оставаться неоднозначным.

Для GeoShturman актуальные продуктовые ограничения смотрите в [GeoShturman/README.md](GeoShturman/README.md), потому что его ядро развивается отдельно.
