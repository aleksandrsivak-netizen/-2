# ГеоШтурман + Теарком — навигация без спутников

Система определяет, **где находится беспилотник и куда он летит, когда GNSS
(GPS/ГЛОНАСС) недоступен** — по форме рельефа под аппаратом и заранее
загруженной карте высот. Сделано под кейс «Полёт вслепую».

Если в двух словах: самолёт знает свою высоту над морем (барометр) и высоту
до земли (радиовысотомер). Вычитаем одно из другого — получаем высоту рельефа
под брюхом в каждый момент. Набегает «профиль местности», как кардиограмма.
Этот профиль ищем на карте высот: перебираем все направления (0–360°) и
положения, считаем корреляцию, берём максимум. Где совпало — там самолёт, и
туда же указывает его курс и скорость.

---

## Главное, что нужно понять про устройство проекта

Проект состоит из **двух самостоятельных частей**, которые лежат в отдельных
папках и **соединены между собой**:

| Папка | Что это | Роль |
|-------|---------|------|
| **`tercom_uav/`** | **Алгоритм** (TERCOM / terrain-referenced navigation) | Считает: координаты, курс, скорость по NMEA-радиовысоте и карте высот. Это «мозг». |
| **`GeoShturman/`** | **Визуализация** (веб-дашборд + API на FastAPI) | Показывает результат: 3D-карта, поток NMEA, курс/скорость, тепловая карта корреляции, метрики. Это «приборная панель». |

**Они связаны так, что дашборд показывает ровно то, что посчитал алгоритм.**
Связь сделана через один тонкий файл-«мост»
(`GeoShturman/backend/app/core/tercom_bridge.py`): визуализация во всех местах
зовёт одну функцию `solve_navigation(...)`, а та по умолчанию передаёт расчёт
в ядро `tercom_uav` и возвращает результат в том формате, который ждёт
дашборд. **Никакой код фронтенда при этом не менялся.**

```
  поток NMEA  ─►  GeoShturman (API/дашборд)  ─►  solve_navigation()
                                                      │
                                                      ▼
                                          tercom_bridge ─► tercom_uav (АЛГОРИТМ)
                                                      │
            дашборд рисует ◄── координаты, курс, скорость, heatmap, confidence
```

Переключатель движка — переменная окружения `NAV_ENGINE`:
- `NAV_ENGINE=tercom` (по умолчанию) — считает алгоритм Теаркома;
- `NAV_ENGINE=native` — считает старый встроенный движок ГеоШтурмана (оставлен
  как запасной и для сравнения).

## Как запустить у себя за 5 минут (проверено на Windows)

Нужен **64-битный Python 3.11+** (на 32-битном не ставятся numpy/scipy).

```powershell
# 1. встать в корень проекта (где лежат папки tercom_uav и GeoShturman)
cd путь\к\tercom-test-clone

# 2. создать виртуальное окружение и поставить зависимости
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install numpy scipy pandas matplotlib typer PyYAML `
    fastapi "uvicorn[standard]" python-multipart pydantic pydantic-settings pillow httpx

# 3. запустить сервер (движок Теаркома включается переменной NAV_ENGINE)
cd GeoShturman\backend
$env:NAV_ENGINE = "tercom"
..\..\.venv\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

Открыть в браузере: **http://127.0.0.1:8000**

> Папку `tercom_uav` **не нужно** ставить отдельно — мост сам подхватывает её
> из соседнего каталога. Главное, чтобы обе папки лежали рядом.

### Что нажимать на дашборде

1. **«Старт»** — запустит живой полёт по рельефу (симуляция). Поедет борт по
   карте, слева побегут строки `$GPGGA`, обновятся курс, скорость, ошибка
   позиции (CEP), построится 3D-поверхность корреляции. Всё это считает Теарком.
2. **«Ввод данных»** — вставить свой NMEA-поток текстом, загрузить файлом или
   задать параметры руками. Кнопка «Решить разом» даст разовую привязку.
3. **3D-карта** крутится мышкой, зум — колесом.
4. **http://127.0.0.1:8000/docs** — Swagger, чтобы дёргать API руками.

### Запасной движок и реальный рельеф

```powershell
$env:NAV_ENGINE = "native"      # сравнить со старым движком
$env:DEM_SOURCE = "real"        # реальный рельеф Хибин из Copernicus GLO-30
                                # (требует: pip install rasterio pyproj)
```

## Что уже проверено (независимыми тестами)

В папке `independent_test/` лежит **внешний строгий стенд**, который проверяет
алгоритм «вслепую»: он сам генерирует рельеф, сам снимает с него высоту своим
кодом, сам кодирует NMEA и **не сообщает алгоритму правильный ответ** —
координаты, курс и скорость алгоритм должен вывести сам.

**Результат на чистом ядре Теаркома (14 сценариев из 14 пройдено):**
- азимут — ошибка **0.00°**, направление по всем квадрантам 0–360° находится точно;
- скорость — ошибка **~1.8%** даже при намеренно искажённой подсказке (скорость
  реально является *выходом* алгоритма, а не входом);
- координаты постфактум — **порядка 5 метров** на трассе ~9 км;
- на **плоском рельефе** алгоритм честно опускает достоверность и поднимает флаг
  «ненадёжно» (а не выдумывает координаты — по рельефу там цепляться не за что).

**Результат проверки связки (один и тот же NMEA через оба движка):**
```
старый движок:  азимут 128.0° (Δ0)  скорость 40.0 (Δ0)  ошибка позиции 0 м   достоверность 0.85
Теарком (мост): азимут 128.0° (Δ0)  скорость 40.0 (Δ0)  ошибка позиции 16 м  достоверность 0.86
расхождение между движками: Δазимут 0°, Δпозиция 16 м  → указывают в одну точку
```

Запустить эти тесты:
```powershell
.\.venv\Scripts\python.exe independent_test\independent_harness.py   # стенд по алгоритму
.\.venv\Scripts\python.exe independent_test\test_consistency.py      # связка двух движков
```

## Фильтры и обработка сигнала (каскад)

Поток NMEA от датчика «грязный», поэтому стоит цепочка очистки и устойчивой оценки:

1. **Хампель / медиана** на входе — выкидывает битые строки и скачки
   радиовысотомера (медиана + MAD). На дашборде виден счётчик отброшенных выбросов.
2. **Фильтр Калмана 1D** — сглаживает профиль высот перед сопоставлением.
3. **Корреляция по рельефу (TERCOM, ядро Теаркома)** — основная привязка:
   перебор всех направлений и сдвигов, поиск максимума корреляции.
4. **Счисление пути (dead reckoning)** — ведёт позицию между привязками к
   рельефу, как инерциалка; компенсирует дрейф барометра.
5. **Самооценка достоверности** — по информативности рельефа и остроте пика
   корреляции; над плоским местом честно проседает.

## Честно об ограничениях

- Алгоритм отлично работает на **прямолинейном** полёте над выраженным рельефом.
  На резких манёврах позицию между привязками держит счисление — для резких
  разворотов нужен более частый пересчёт.
- Полный перебор сетки точный, но тяжёлый для слабого бортового вычислителя —
  под реальную плату нужна оптимизация (FFT-корреляция / Numba).
- Над **плоской** местностью точность физически падает (рельефа мало). Система
  это видит сама и честно сигнализирует, а не делает вид, что всё хорошо.
- По умолчанию карта **синтетическая** (для воспроизводимости и быстрого
  старта). Реальный рельеф Copernicus включается `DEM_SOURCE=real`.

## Развёртывание в Docker (нюанс)

`docker compose` в папке `GeoShturman` собирает образ только из неё, поэтому
соседняя папка `tercom_uav` в образ не попадёт. Чтобы прод-сборка тоже работала
на Теаркоме, нужно при сборке **скопировать `tercom_uav` в образ** (расширить
build-context до корня репозитория) или поставить его как пакет
(`pip install ./tercom_uav`). Локальный запуск через `uvicorn` из корня этого
не требует — мост находит соседнюю папку сам.
